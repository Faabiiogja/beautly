🏗️ Estratégia Multi-tenant por Subdomínio

// middleware.ts — roda no Edge (zero cold start)

export function middleware(req: NextRequest) {
  const host = req.headers.get('host') ?? ''

  // 1. Super Admin
  if (host === 'admin.beautly.com') {
    return handleSuperAdmin(req)
  }

  // 2. Dev local — usa tenant do .env
  if (host.includes('localhost') || host.includes('127.0.0.1')) {
    slug = process.env.DEV_TENANT_SLUG ?? 'demo'
  }

  // 3. Preview Vercel (*.vercel.app) — usa tenant do .env
  else if (host.includes('vercel.app')) {
    slug = process.env.PREVIEW_TENANT_SLUG ?? 'demo'
  }

  // 4. Produção — extrai slug do subdomínio
  else {
    slug = host.split('.')[0] // "clinica" de "clinica.beautly.com"
  }

  // Injeta slug no header — Server Components leem com headers()
  const res = NextResponse.next()
  res.headers.set('x-tenant-slug', slug)
  return res
}
⚠️ O middleware não consulta o banco — só repassa o slug. A resolução real (slug → tenant) acontece uma vez no layout raiz com cache. Isso mantém o Edge ultra-rápido.

🔒 Autenticação — 3 contextos

👑 Super Admin (você)
Credencial única em .env
SUPER_ADMIN_EMAIL
SUPER_ADMIN_PASSWORD_HASH

Login via /api/auth/super
Gera JWT com claim role: "super"
Cookie httpOnly + secure
Sem Supabase Auth
Simples e suficiente para 1 usuário. Quando precisar de equipe, migra para Supabase.
🔑 Tenant Admin
Supabase Auth
Email + senha

Login: supabase.auth.signInWithPassword()
Sessão via cookie Supabase
Middleware valida sessão
Verifica tenant_users.tenant_id
redireciona se não autorizado
Tenant A não acessa painel do Tenant B — validado no middleware + RLS.
🌸 Cliente Final
Sem autenticação tradicional

Acesso via token único na URL:
/b/[token]

Token → booking_tokens
Verifica: existe? expirou? revogado?
Cookie opcional para UX
(lembra o token no mesmo dispositivo)
Nunca expõe phone na URL. Nunca permite login só por telefone.

🔗 Link Seguro do Cliente — ciclo completo

1. Geração token = crypto.randomBytes(32).toString('hex') (64 chars, 256 bits de entropia)
expires_at = now() + 7 dias
INSERT booking_tokens 2. URL https://clinica.beautly.com/b/a3f9c2...d8e1
Token nunca contém booking_id, nome ou telefone 3. WhatsApp Meta Cloud API envia template aprovado com o link
"Seu agendamento está confirmado! Acesse: [link]" 4. Validação SELECT * FROM booking_tokens WHERE token = $1
AND expires_at > now() AND revoked_at IS NULL
→ retorna booking completo via JOIN 5. Ação Cancelamento: booking.status = 'cancelled' + token.revoked_at = now()
Reagendamento: novo booking + novo token + revoga token anterior 6. Cookie httpOnly cookie com o token (7 dias)
Na próxima visita no mesmo dispositivo: redireciona direto para /b/[token]

🛡️ Segurança mínima para produção

✅ Implementar no MVP
• RLS no Supabase em todas as tabelas
• tenant_id em toda query de aplicação
• HTTPS obrigatório (Vercel cuida)
• Cookies httpOnly + secure + sameSite=lax
• Token de 256 bits (inviável brute force)
• Rate limit em /api/bookings (ex: 10 req/min por IP)
• Validar/sanitizar inputs com Zod
• Headers de segurança via next.config.js
• Variáveis sensíveis só no .env (nunca no código)
⏳ Depois do MVP
• OTP por WhatsApp/SMS
• 2FA para admin
• Audit log de ações admin
• WAF (Cloudflare)
• Monitoramento de anomalias
• LGPD: consentimento e exclusão de dados
• Webhook signatures para Meta API

Anti-enumeração de telefone

O fluxo de agendamento nunca pergunta telefone para consultar agendamento.
O cliente acessa apenas pelo link que recebeu no WhatsApp.
Não existe endpoint público GET /bookings?phone=xxx.
O campo phone é nunca retornado em responses públicas — só usado internamente para upsert de customer.
Rate limit de 5 tentativas/min em qualquer endpoint que aceite phone como input.
