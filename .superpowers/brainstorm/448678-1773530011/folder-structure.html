app/ — só UI e roteamento
Pages e layouts. Nada de lógica de negócio aqui.
Route groups (booking), (admin), (super-admin) organizam sem afetar a URL.
domain/ — lógica de negócio
Cada domínio tem: queries.ts (leitura), actions.ts (escrita via Server Actions), types.ts.
Testável independente da UI. Fácil de extrair se precisar.
lib/ — infraestrutura e integrações
Supabase, tokens, disponibilidade, WhatsApp.
Sem acoplamento a React ou Next.js — puro TypeScript.
validations/ — Zod em toda borda
Schemas compartilhados entre form (client) e Server Action (server).
Um schema, dois usos. Nunca validação duplicada.
Fluxo de dados
page.tsx chama domain/*/queries.ts
Forms chamam domain/*/actions.ts (Server Actions)
Actions usam lib/ para integrações externas
Nada de prop drilling — dados no server, estado mínimo no client
